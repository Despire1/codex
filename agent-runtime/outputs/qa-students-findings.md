# QA Аудит: раздел «Ученики»

Дата: 2026-04-28
Тестировщик: Claude (Local Teacher)
Скоуп: /students — список, карточка ученика, CRUD, связанные сущности.

## Findings (черновик до переноса в Linear)

### B1. Сайдбар «прилипает» в развёрнутом виде и перекрывает контент
- **Где:** глобально, любая страница, hover на левый рельс.
- **Воспроизведение:** навести мышь на левый рельс → меню разворачивается. После клика по пункту меню оно остаётся раскрытым и перекрывает заголовок и контент. Сворачивается только при клике в стороне.
- **Ожидание:** при mouseleave / навигации сайдбар должен сворачиваться обратно.
- **Severity:** Major.

### B2. Опечатка / странный username «@usernmae3» в seed-данных
- **Где:** карточка ученика «Петр Полтавский» → Информация.
- **Замечание:** похоже на опечатку (`usernmae3` вместо `username3`). Если username вводится вручную — нужна валидация формата Telegram (5-32 символа, латиница/цифры/_).

### B3. В шапке карточки чип «Telegram не привязан» при наличии username
- **Где:** /students/:id шапка.
- **Проблема:** показывается «Telegram не привязан», но в блоке «Информация» отображается @username.
- **Гипотеза:** username указан вручную, реальная привязка через telegramId ещё не произошла. Чип читается двусмысленно. Стоит переименовать в «Не подключился к боту» или вынести из ряда статусов.

### B4. KPI «5 показателей в шапке» — нагромождение
- **Где:** /students/:id, верхний ряд (Уроков проведено / Цена за урок / Средний балл / Занятия на балансе / Домашек сдано).
- **UX:** для пустого ученика 4 из 5 значений «0», воспринимается как ошибка/пустота. Предложение: для нулевых значений показывать прочерк или скрывать средний балл, если уроков 0.

### B5. Critical: нет действий «Поставить на паузу» и «Завершить обучение»
- **Где:** /students/:id шапка → меню «⋯».
- **Проблема:** меню содержит только «Редактировать ученика», «Настройки уведомлений», «Удалить ученика». Нет действий, которые упоминаются в пустом состоянии: пустые табы «На паузе» / «Завершили» гласят: «Поставить на паузу можно из профиля ученика через меню «⋯»» — этого пункта в меню нет.
- **Воспроизведение:** /students → открыть любого ученика → ⋯ в правом верхнем углу карточки.
- **Severity:** Critical (фича недоступна; UI обещает функциональность, которой нет).

### B6. Кнопка «Дополнительные фильтры» (иконка воронки) ничего не делает
- **Где:** /students, тулбар фильтров, между сортировкой и переключателем вида.
- **Проблема:** клик по кнопке не открывает попап/панель и не меняет состояние списка.
- **Severity:** Major (мёртвая кнопка в основном тулбаре).

### B7. Сортировка «По успеваемости» работает некорректно
- **Где:** /students, тулбар → Сортировка → По успеваемости.
- **Проблема:** ученик «Сергей» с 11 уроками и 100% посещаемостью оказывается на 7-м (последнем) месте, после учеников с 0 уроков и нулевой посещаемостью. Также Artem (100%, 29 уроков) выше Сергея (100%, 11 уроков) — это ок, но позиция Сергея ниже нулевых учеников — баг.
- **Гипотеза:** в сортировке не учитывается attendance (или используется другое поле, отличное от того, что показано в столбце «Посещаемость»). Скорее всего, сортируется по avgScore, а у Сергея avgScore не выставлен.
- **Severity:** Major.

### B8. Список уровней в фильтре «Все уровни» — distinct по существующим студентам, а не справочник
- **Где:** /students → фильтр «Все уровни».
- **Поведение:** показывает только уровни, которые уже выбраны хотя бы у одного ученика (B2, HSK2-3). Уровни A1, A2, B1, C1, C2, HSK1, HSK4-6 в выпадайке отсутствуют, даже если поддерживаются формой создания.
- **Замечание:** Если задумано так — ок (UX-выбор). Но в идеале нужно показывать стандартный справочник.
- **Severity:** Minor.

### B9. Сайдбар не имеет визуальной разделительной линии между «Помощь» и нижней зоной
- **Где:** Левый рельс.
- **Замечание:** из «Развернуть» внизу нет hover-стейта, не очевидно, что это кнопка.
- **Severity:** Trivial.

### B10. Форма «Добавить ученика»: молчаливая блокировка submit при отрицательной/нулевой цене
- **Где:** /students → «Добавить ученика» → Цена занятия.
- **Проблема:** ввод «-100» или «0» — submit ничего не делает, нет toast/inline-сообщения. Только тонкая зелёная (!) рамка фокуса. Пользователь не понимает, что мешает.
- **Severity:** Major (UX, рискует «зависшей» формой).

### B11. Валидация имени через модалку поверх модалки — анти-паттерн
- **Где:** /students → «Добавить ученика», submit без имени (или из пробелов).
- **Проблема:** показывается модальный диалог «Заполните все поля → Укажите имя ученика» с кнопкой «Понятно». Это вторая модалка поверх первой и нарушает best practice — для одного отсутствующего поля нужна inline-error прямо под input.
- **Severity:** Minor.

### B12. Поле «Уровень ученика» — свободный текст, нет справочника/автокомплита
- **Где:** /students → форма создания/редактирования.
- **Проблема:** можно ввести «бла-бла», эмодзи, любую строку. Это засоряет фильтр «Все уровни» (B8) и нарушает консистентность.
- **Предложение:** combobox с пресетами CEFR (A1–C2) и HSK (1–6) + возможность кастомного значения.
- **Severity:** Minor.

### B13. Невалидный e-mail при редактировании молча отклоняется
- **Где:** /students/:id → Редактировать → Email.
- **Проблема:** ввод «not-an-email» → submit ничего не делает (HTML5 type=email блокирует), но нет ни тоста, ни визуального сообщения.
- **Severity:** Minor.

### B14. У первой заметки (созданной из формы создания ученика) нет timestamp
- **Где:** /students/:id → блок Заметки.
- **Проблема:** заметка, заполненная в форме создания (поле «Заметки»), отображается с подписью «Из карточки ученика» вместо даты/времени. У всех остальных заметок есть «28 апреля 2026 • 19:41».
- **Severity:** Minor.

### B15. После сохранения редактирования ученика нет toast «Сохранено»
- **Где:** /students/:id → Редактировать ученика → сохранить.
- **Проблема:** PATCH 200, данные обновлены — но никакой обратной связи. У других действий (toggle напоминаний, удаление) toast есть.
- **Severity:** Minor (консистентность).

### B16. **Critical:** «Восстановление при повторном добавлении» работает частично и привносит непрошеные побочные эффекты
- **Где:** /students/:id удалить → создать снова с тем же telegram username.
- **Текст подтверждения:** «Связь с учеником будет удалена из вашего списка. Данные ученика сохранятся и восстановятся при повторном добавлении.»
- **Что произошло на практике (id 15, telegram=qa_bot_new):**
  - ✅ Сохранились: id ученика, проведённые/назначенные уроки.
  - ❌ Сбросились: цена занятия (1700 → дефолт 1600), уровень (B2), email, телефон, заметки.
  - ❌ Применилось имя из новой формы (что ок, но не «восстановилось»).
  - ❌ Появился чип «Ученик на паузе», хотя я никогда его не ставил, и интерфейса для этого статуса нет (см. B5).
- **Severity:** Critical (документация-обещание не соответствует поведению; данные теряются молча; появляется фантомный статус).

### B17. KPI «Уроков проведено = 1» при единственном «Запланированном» уроке
- **Где:** /students/:id, KPI «Уроков проведено».
- **Проблема:** созданный сегодня урок 28 апр 18:00–19:00 со статусом «Запланирован» считается проведённым (KPI показал 1), хотя не имеет статуса «Завершён»/«Проведён».
- **Гипотеза:** счётчик считает «уроки, время которых уже прошло», а не реальный статус. Сейчас 19:41 — урок до 19:00, поэтому уже прошёл.
- **Severity:** Major (расходится с пониманием статусов).

### B18. Цель занятий не имеет UI редактирования/удаления вне формы
- **Где:** /students/:id → блок «Цель занятий».
- **Проблема:** при hover не появляются иконки edit/delete. Чтобы изменить цель — надо открывать форму редактирования ученика. Заметки же редактируются inline.
- **Severity:** Minor (несогласованность с заметками).

### B19. Иконки edit/delete у «системной» заметки выглядят disabled, но кликабельны
- **Где:** /students/:id → первая заметка (с подписью «Из карточки ученика»).
- **Проблема:** при hover иконки появляются полупрозрачными (как disabled) → пользователь думает, что заметку нельзя редактировать. На самом деле клик работает и подтверждение удаления тоже.
- **Severity:** Minor (визуальная неоднозначность).

### B20. После очистки опциональных полей блок «Информация» не показывает уровень, если он есть
- Подтверждено в ходе теста, что блок «Информация» корректно скрывает email/телефон, если они пустые. Уровень тоже скрывается, если пуст. Это скорее ОК, но: если нет ни email, ни телефона, ни уровня и Telegram unmapped — блок «Информация» состоит только из @username и даты. Можно показывать placeholder типа «Заполните дополнительную информацию».
- **Severity:** Trivial (улучшение).

### B21. Сортировка «По имени» не строго алфавитная (русский → латиница)
- **Где:** /students → Сортировка: По имени.
- **Поведение:** Антон, Иван, Петр, Полина, Сергей, **Artem, QA Тестовый Ученик** — латиница оказывается в конце (collate ru_RU). Это нормально, но «Artem» — единственный латинский ученик с реальной активностью; пользователю может быть неочевидно, где его искать. Возможно стоит давать выбор «по локали» или «единый Unicode-collate».
- **Severity:** Trivial.

### B22. Telegram username принимается с пробелом, который молча убирается, без обратной связи
- **Где:** /students → форма «Добавить ученика», поле Telegram username.
- **Поведение:** ввод «invalid space» → input стал «invalidspace» (пробелы вырезаны). Хорошо, что нормализация есть, но пользователь не понимает, что произошло — стоит показать подсказку «Пробелы недопустимы — удалены».
- **Severity:** Trivial.

### B23. XSS-инъекция в имя — экранируется (✓ — позитивная проверка)
- **Где:** форма создания, поле «Имя».
- **Тест:** ввёл `QA Edge Тест <script>alert(1)</script>` → не выполнено как HTML. Сохранён как plain text (затем перезаписан другим именем).
- **Статус:** OK, проблем нет.
